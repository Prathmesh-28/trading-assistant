# BACKLOG — basic-features audit (600+ items)

Companion to [ROADMAP.md](ROADMAP.md) (strategic features) and [STRATEGIES.md](STRATEGIES.md) (trading logic). This is the unglamorous engineering/product debt: every basic thing a professional reviewer would flag as missing or shallow. Generated by a 10-agent parallel audit, deduplicated.

Note: several §1 items were fixed the same day this audit ran (login + token auth on API and WebSocket, login rate-limit/lockout, demo/live banner + landing page) — kept for completeness.

## Authentication, sessions & security (62 items)

1. **Hardcoded default password in config.py** — config.py ships AUTH_PASSWORD default 'trade@2026', so a Render deploy without the env var is instantly guessable and exposes live trade positions and Groww-linked state.
2. **Plaintext password comparison, no hashing** — The single password is stored/compared as a plain string instead of bcrypt/argon2, so any log leak, env dump, or repo clone reveals the credential directly.
3. **No logout endpoint** — There is no server-side /logout that invalidates the session token, so a token stolen from a shared or public machine stays valid indefinitely for a tool that shows live P&L and stops.
4. **No session expiry / token TTL** — Issued tokens never expire, meaning one successful login on a phone browser grants permanent access to trade-confirmation commands like /bought and /sold.
5. **No token revocation on password change** — Even if the password env var is rotated, previously issued tokens keep working because there is no server-side session store or token versioning.
6. **No password change flow** — Changing the password requires editing Render env vars and redeploying; there is no authenticated change-password endpoint, so rotation after a suspected leak is slow during market hours.
7. **No password reset / recovery** — A forgotten password locks the user out of monitoring live positions mid-trade until a redeploy, with no email/OTP recovery path.
8. **No login rate limiting** — The login endpoint accepts unlimited attempts, so the single short password can be brute-forced against the public Render URL in hours.
9. **No account lockout or backoff** — There is no exponential delay or temporary lockout after repeated failures, compounding the brute-force exposure of a one-account system.
10. **No CAPTCHA or proof-of-work on login** — The public landing page login can be hammered by bots with zero friction, and Render free tier will also burn CPU serving those attempts.
11. **No audit log of logins** — Successful/failed logins are not written to the SQLite journal, so the user cannot detect that someone else has been viewing or confirming trades.
12. **No alert on failed login attempts** — The app already has a Telegram notifier but never uses it to warn about repeated failed logins on a tool that controls stop management.
13. **No alert on new-device login** — A login from an unfamiliar IP/user-agent triggers no Telegram notification, so a compromised credential goes unnoticed.
14. **WebSocket endpoint unauthenticated** — The /ws snapshot stream broadcasts positions, qty, and stops to any connected client without checking the login token.
15. **REST command endpoints lack per-request auth check** — Mutating endpoints that route into RecommendEngine.handle_command() (bought/sold/exit) are not consistently gated by the token, letting anyone move stops via curl.
16. **Token sent as query param or localStorage** — The React app stores the token in localStorage rather than an HttpOnly cookie, so any XSS in the dashboard exfiltrates full account access.
17. **No HttpOnly/Secure/SameSite cookie session option** — Without cookie-based sessions with SameSite=Lax, the API is exposed to both XSS token theft and CSRF from other tabs.
18. **No CSRF protection on mutating endpoints** — If cookies are ever used, POSTs like /command/exit have no CSRF token, so a malicious page could force-close a live position.
19. **CORS likely wildcarded** — The Render backend must allow the Vercel origin, but a `*` or overly broad CORS config lets any website's JS call the trade API with a leaked token.
20. **No security headers middleware** — Responses lack X-Content-Type-Options, X-Frame-Options/frame-ancestors, and Referrer-Policy, leaving the dashboard clickjackable and sniffable.
21. **No Content-Security-Policy on dashboard** — The Vercel-served React app has no CSP, so a single injected script (e.g., via a symbol name rendered unsafely) can read the auth token and positions.
22. **No HSTS header** — Neither Render nor the app sets Strict-Transport-Security, allowing downgrade attacks on a session that carries trading data.
23. **No HTTPS-only redirect enforcement in app** — The app does not reject or redirect plain-HTTP requests itself, relying entirely on platform defaults that can change.
24. **Single shared account, no user model** — There is exactly one username with no user table, so credentials cannot be scoped, disabled individually, or extended to a read-only viewer for a spouse/accountant.
25. **No role separation (view vs trade-confirm)** — Anyone who logs in can confirm fills and exit positions; there is no read-only mode for checking the dashboard from an untrusted machine.
26. **No 2FA/TOTP on dashboard login** — Ironically the app stores a GROWW_TOTP_SECRET for the broker, but its own login protecting the same positions is single-factor password only.
27. **No remember-me vs short-session distinction** — There is one implicit session behavior instead of a 'keep me signed in' choice, so mobile convenience and shared-desktop safety cannot both be served.
28. **No idle-timeout auto-logout in frontend** — The dashboard never locks after inactivity, so a screen left open at work continuously shows live positions and accepts stop changes.
29. **No concurrent-session listing or kill switch** — The user cannot see active sessions or remotely revoke one, which matters when logging in from cybercafe/office machines to check trades.
30. **Telegram bot has no chat-ID allowlist enforcement audit** — If /bought and /sold commands are accepted from any chat rather than a strictly verified TELEGRAM_CHAT_ID, anyone discovering the bot can mark trades filled.
31. **No signing/verification of Telegram webhook or polling source** — There is no secret-token check on Telegram updates, so spoofed updates could inject engine commands that mutate stops.
32. **Groww API key and TOTP secret in plain .env** — Broker credentials sit unencrypted in .env/Render env with no secret-manager integration or at-rest encryption, and a single misconfigured endpoint could echo them.
33. **.env.example may include real-looking secrets** — .env.example is modified in the working tree; without a check it is easy to accidentally commit a real AUTH_PASSWORD or Groww key to the public repo.
34. **No secret rotation procedure or docs** — ONBOARDING.md has no runbook for rotating Groww/Telegram/auth secrets after a laptop loss, which for broker credentials is a real-money risk.
35. **Secrets can leak into logs** — There is no log scrubbing/redaction layer, so a debug print of config or a failed-request dump on Render can expose the Groww TOTP secret.
36. **Error responses may leak stack traces** — FastAPI default 500 handling (or debug mode) can return tracebacks revealing file paths, config values, and internal logic to unauthenticated callers.
37. **No request logging with IP/user-agent** — There is no access log tying API calls to IPs, so post-incident forensics on who exited a position is impossible.
38. **No audit trail for mutating commands** — The SQLite journal records trades but not WHO/what-session issued each /exit or stop change, so a hijacked session's actions are indistinguishable from the user's.
39. **No generic-error on failed login** — If login errors distinguish 'bad username' from 'bad password', they confirm the valid username ('prathmesh' is also the default in the repo).
40. **Username default committed to public repo** — AUTH_USERNAME defaults to the owner's real first name in config.py, halving the brute-force problem to password-only.
41. **No password strength policy or validation** — Nothing prevents setting a 4-character AUTH_PASSWORD, and the shipped default 'trade@2026' would fail any basic strength check.
42. **No timing-safe credential comparison** — Password check likely uses `==` instead of hmac.compare_digest, enabling timing side-channel attacks against the public endpoint.
43. **JWT/token signing secret defaulting or weak** — If tokens are signed, the signing key likely has a hardcoded default like the password, letting anyone forge admin tokens offline.
44. **No token audience/issuer/claims validation** — Tokens carry no scoped claims (exp, iat, purpose), so a token minted for viewing works forever for every mutating command.
45. **No API rate limiting on data endpoints** — Snapshot/quotes endpoints have no throttle, so scraping or an accidental frontend retry loop can exhaust Render free-tier and Groww quota, blinding live stop monitoring.
46. **No rate limit on Telegram command processing** — A flood of bot commands isn't throttled, and each mutating command triggers broadcasts, allowing trivial DoS of the engine loop during market hours.
47. **Health endpoint exposes internal state unauthenticated** — Status/health routes that report engine mode, symbols, or creds-configured flags leak reconnaissance data (e.g., the error string mentioning GROWW_API_KEY setup).
48. **No dependency vulnerability scanning** — requirements.txt is unpinned/unaudited with no pip-audit or Dependabot, risky for a service holding broker credentials.
49. **Frontend build embeds backend URL without env gating** — If any token or API secret is referenced via VITE_ vars, it is baked into the public JS bundle on Vercel.
50. **No session restore hardening after backend restart** — Render free tier restarts frequently; if auth state is in memory, restarts either invalidate sessions silently (bad UX mid-trade) or skip auth entirely (bad security), and neither is handled explicitly.
51. **No brute-force protection on WebSocket reconnects** — Unlimited WS reconnect attempts with bad/absent tokens are neither rejected early nor rate-limited, wasting the single free-tier worker.
52. **SQLite journal file has no access protection** — journal.db permissions and path are unmanaged; on a shared host or leaked backup it exposes full trade history and sizes with no encryption.
53. **No backup encryption or exclusion for journal** — If the SQLite journal is ever synced/committed for persistence (common on ephemeral Render disks), the trading history goes to GitHub unencrypted.
54. **Login page lacks autocomplete/password-manager hints** — Missing autocomplete='current-password' and proper form semantics discourages password-manager use, pushing the user toward a weak memorable password.
55. **No 401 handling / auto-redirect in dashboard** — When a token expires or is rejected, the React app likely shows broken data instead of redirecting to login, and may silently retry with the dead token forever.
56. **No last-login display after sign-in** — The dashboard never shows 'last login at X from Y', the cheapest way for a solo trader to notice account compromise.
57. **No IP allowlist option for mutating routes** — For a single-user tool, an optional home/mobile IP allowlist on trade-confirmation endpoints is cheap defense-in-depth but absent.
58. **No graceful auth-disabled mode flagging** — If AUTH_USERNAME/PASSWORD are unset there is no explicit fail-closed behavior or startup warning; the fail-soft philosophy risks the API silently running open.
59. **No protection of backtest endpoint with real creds** — Backtest runs use real Groww creds; if exposed via API without auth, an attacker can burn broker API quota and infer the account's data entitlements.
60. **No token binding to user-agent/IP heuristics** — Stolen tokens work from anywhere with no soft checks or re-auth prompts on drastic context change, unusual for an app controlling money decisions.
61. **No re-authentication for destructive actions** — Exiting all positions or clearing state requires no password re-entry, so one hijacked dashboard tab can flatten the book.
62. **No security section in ONBOARDING.md** — The onboarding doc covers code tour but not the threat model, auth design, or secret handling, so future changes (e.g., new endpoints) will forget to gate them.

## UX states (loading / empty / error / offline) (61 items)

63. **No WebSocket disconnected banner** — When the WS to Render drops, the dashboard silently keeps showing the last snapshot with no 'disconnected — reconnecting' banner, so the user may act on frozen prices for a live position.
64. **No last-updated timestamp on prices** — LTP, P&L, and stop distances show no 'as of HH:MM:SS' stamp, so during Render cold starts or feed stalls the user cannot tell live from stale quotes.
65. **No stale-quote visual dimming** — Quotes older than N seconds are not greyed or flagged, which is dangerous when a Chandelier trail decision hinges on the current tick.
66. **Reconnect retries lack backoff status UI** — useLive retries every 2s forever with no visible attempt counter or 'backend unreachable' escalation, leaving the user staring at a spinner during Render free-tier sleep.
67. **No Render cold-start explainer state** — Free-tier Render takes 30-60s to wake; the loading screen has no 'server waking up, ~1 min' message, so the app just looks broken.
68. **No confirmation dialog on Exit tap** — Tapping Exit on a PositionCard immediately mutates engine state with no 'Exit RELIANCE at ~2450?' confirm, risking accidental position closure from a mis-tap.
69. **No confirmation echo on /bought fills** — Confirming a buy does not restate qty/price for review before committing, so a fat-fingered fill price silently corrupts P&L and stop math.
70. **No undo after confirming a trade** — A mistaken /bought or /sold cannot be reverted from the UI; the user must edit SQLite by hand, unacceptable for a manual-entry workflow.
71. **No optimistic update on idea actions** — Tapping an IdeaCard action waits for the full round trip with no immediate pending state, so on slow Render the button seems dead and gets tapped twice.
72. **No double-submit guard on action buttons** — Buttons are not disabled while a command is in flight, so two quick taps can send /bought twice and double-count a position.
73. **No failure rollback for optimistic actions** — If a mutation is added optimistically and the POST fails, there is no rollback plus error toast, leaving UI state diverged from engine state.
74. **No toast system at all** — Command results (accepted, rejected by heat cap, symbol unknown) have no transient toast feedback; the user must infer success from the next snapshot.
75. **Alerts use one banner slot only** — AlertBanner shows a single message, so a stop-hit alert can be overwritten by a routine break-even note before the user sees it.
76. **No alert acknowledgement/dismiss persistence** — Dismissed alerts reappear on refresh because acknowledgement is not stored, training the user to ignore banners including stop-hit ones.
77. **No distinct empty state for no-ideas days** — An empty ideas list looks identical to a failed fetch; it should say 'engine ran at 09:20, no setups passed the regime gate today'.
78. **No empty state for zero positions** — The positions panel renders blank instead of 'No open positions — heat available ₹X', missing a chance to surface risk capacity.
79. **No empty state for history table** — HistoryTable shows nothing for a fresh journal instead of 'No closed trades yet', which reads as a data bug.
80. **No skeleton loaders for cards** — Idea/position cards pop in after the snapshot arrives with layout shift; skeleton placeholders would signal loading and prevent mis-taps.
81. **BacktestPanel has no progress indicator** — A 730-day backtest on free-tier Render can take tens of seconds with no progress bar or 'fetching candles 3/10 symbols' status, so users cancel or re-submit.
82. **No backtest failure reason surfaced** — When Groww creds are missing or candles unavailable, the backtest just returns nothing instead of 'historical data unavailable: credential error'.
83. **No timeout/cancel affordance for backtests** — There is no cancel button or client timeout for a hung backtest request, so the panel can spin forever.
84. **Chart panel lacks no-data state** — ChartPanel for a symbol with no cached candles renders empty axes instead of 'no intraday data for MUTHOOTFIN', which looks like a rendering bug.
85. **No retry button on failed data fetches** — Failed REST calls (history, backtest, chart) require a full page reload; every error state should offer an inline Retry.
86. **No global error boundary in React** — One malformed field in a snapshot (types.ts drift) can white-screen the whole dashboard instead of isolating the broken card with a fallback.
87. **No handling for snapshot schema mismatch** — If the deployed backend and Vercel frontend versions drift, the app should show 'update available / incompatible data' instead of undefined-field crashes.
88. **No market-closed idle state** — Outside 09:15–15:30 IST the dashboard should switch to a clear 'market closed, next session opens in 16h' mode rather than showing stale intraday panels.
89. **No holiday-aware market status** — MarketStatusBadge does not distinguish NSE holidays from weekends or a dead feed, so a silent feed outage on a trading day looks like 'closed'.
90. **No degraded-mode indicator for missing creds** — The engine fails soft when Groww creds are absent, but the dashboard never says 'running without live data', hiding that ideas are impossible.
91. **No Telegram-outage indicator on dashboard** — If notifier fails, stop-hit alerts silently stop reaching the phone with no 'Telegram delivery failing since 10:42' warning on the dashboard.
92. **No engine heartbeat / last-scan indicator** — There is no 'engine last ran 14:32:05' tile, so a crashed or stalled engine loop is indistinguishable from a quiet market.
93. **No offline detection in browser** — navigator.onLine / offline events are unused, so on mobile with dropped connectivity the app shows live-looking numbers instead of an offline overlay.
94. **No stale-on-restore flag after boot** — After state restore from SQLite on server boot, positions render as if freshly monitored with no 'restored from journal, prices not yet refreshed' badge.
95. **Login form lacks inline validation messages** — The new username/password form does not show field-level errors (empty password, bad username) before submit, only a generic failure.
96. **No specific wrong-credentials error message** — Login failures should distinguish 'invalid credentials' from 'server unreachable', since Render sleep makes network failures common.
97. **No loading state on login submit button** — The login button has no spinner/disabled state during the auth round trip, so users retap during Render wake-up and queue duplicate requests.
98. **No session-expired handling** — When the auth token expires mid-session, API calls presumably fail silently; the app should intercept 401s and show a 're-login required' modal preserving context.
99. **No caps-lock or show-password affordance** — The password field lacks a visibility toggle and caps-lock hint, a basic form nicety that reduces lockouts on a mobile trading dashboard.
100. **No rate-limit or lockout feedback on login** — Repeated failed logins give no 'try again in 30s' message, so the user cannot tell throttling from a broken backend.
101. **Telegram commands lack error replies for bad args** — /bought with a malformed price or unknown symbol should reply with the exact parse error and expected format, not silence or a generic failure.
102. **No Telegram confirmation keyboard for destructive commands** — /exit and /flat should present an inline Yes/No keyboard instead of executing immediately from a one-line chat message.
103. **No pending-command spinner state in Telegram** — During Render cold start a Telegram command gets no 'working…' acknowledgement, so users resend and double-execute mutations.
104. **No idempotency on repeated commands** — Sending /bought twice (Telegram retry or double tap) is not deduplicated with a 'already confirmed at 10:31' reply, corrupting position qty.
105. **No validation on manual fill price sanity** — A /bought price 20% away from LTP should trigger a 'price looks off, confirm?' warning instead of being accepted as-is into risk math.
106. **No qty-mismatch warning on confirmation** — If the user reports a qty different from the recommended size, the UI/bot accepts it silently instead of flagging the changed risk per trade.
107. **No expiry countdown on idea cards** — Intraday ORB ideas go stale within minutes but cards show no 'valid until 10:05' countdown or auto-expired greyed state, inviting late chases.
108. **Expired ideas not visually distinct** — Ideas invalidated by price moving past entry still look actionable instead of being struck through with 'entry zone missed'.
109. **No 'why rejected' state for gated ideas** — When the regime or index gate suppresses all ideas, the UI shows nothing rather than 'ideas suppressed: NIFTY below 200EMA', eroding trust in the engine.
110. **No heat-cap-blocked explanation in UI** — An idea blocked by portfolio heat should render disabled with 'blocked: heat 5.8%/6% cap', not simply vanish from the feed.
111. **Watchlist add lacks symbol validation feedback** — Typing an invalid NSE symbol into the Watchlist gives no inline 'symbol not found on NSE' error, leaving a dead row that never gets quotes.
112. **No duplicate-symbol guard message in watchlist** — Adding an already-watched symbol should show a friendly notice instead of silently succeeding or duplicating rows.
113. **No pull-to-refresh or manual refresh button** — When the WS is flaky on mobile there is no explicit refresh control to force a snapshot re-fetch, only a full browser reload.
114. **No visual flash on price/P&L updates** — Live numbers change with no green/red tick flash, so the user cannot tell an updating feed from a frozen one at a glance.
115. **Stat tiles have no loading or n/a state** — StatTile presumably renders 0 or NaN before data arrives, and ₹0 P&L is indistinguishable from 'not loaded yet'.
116. **Sparklines render misleadingly with partial data** — Sparkline with 2-3 warmup points draws a line implying a trend instead of showing an 'insufficient data' placeholder.
117. **No toast vs banner severity hierarchy** — Stop-hit (critical, persistent banner) and 'snapshot refreshed' (transient toast) need different treatments; currently all feedback shares one channel or none.
118. **No sound/vibration option for critical alerts** — A stop-hit while the tab is in background gives no audio cue or browser notification permission prompt, defeating the monitoring purpose.
119. **No confirmation before running expensive backtests** — BacktestPanel fires immediately with no 'this will fetch 730 days for 15 symbols (~60s)' warning, causing accidental heavy runs on free-tier compute.
120. **Form inputs accept invalid backtest parameters silently** — Negative days, zero capital, or a future date range in BacktestPanel should show inline validation errors before submission rather than a server 422 or empty result.
121. **No clock-skew warning for signal timestamps** — If the client clock and IST server time diverge, countdowns and 'as of' labels mislead; a skew check with a warning banner is missing.
122. **No partial-data indicator on snapshot** — When some symbols fail to quote, the snapshot renders normally instead of marking affected cards 'quote unavailable', hiding per-symbol feed gaps.
123. **No dirty-state guard on unsent dashboard input** — Navigating tabs or refreshing while a backtest form or watchlist edit is half-filled discards it with no 'unsaved changes' prompt.

## Settings & configuration UI (63 items)

124. **No settings screen at all in dashboard** — Every tunable (risk %, capital, watchlist) lives only in env vars/config.py, so changing anything requires a Render redeploy instead of a UI edit.
125. **Watchlist not editable from UI** — Adding or removing an NSE symbol should be a dashboard action, not a code/env change, since the watchlist is the single most-touched setting in a signal tool.
126. **No symbol validation on watchlist entries** — A typo like RELIENCE should be rejected against the NSE instrument master at save time, not silently produce no data or engine errors at scan time.
127. **No per-symbol enable/disable toggle** — Users need to mute a symbol temporarily (earnings week) without deleting it and losing its place in the list.
128. **Risk-per-trade % not user-editable** — The core position-sizing input is hardcoded/env-based; users must be able to change 1% to 0.5% instantly when drawdown grows.
129. **Trading capital not editable or updateable** — Capital drifts with PnL and deposits, so a stale capital figure makes every qty suggestion wrong until a redeploy.
130. **No portfolio heat cap setting in UI** — Max concurrent open-risk is a personal risk preference that should be adjustable without touching config.py.
131. **No max-positions / max-ideas-per-day setting** — Users can't cap how many simultaneous signals they receive, leading to overtrading on busy days.
132. **No settings validation with typed bounds** — Nothing prevents risk %=50 or capital=-1; settings need range checks (e.g. risk 0.1-5%) with clear error messages before persisting.
133. **No cross-field validation of settings** — E.g. per-trade risk exceeding portfolio heat cap, or min qty times price exceeding capital, should be flagged at save, not at signal time.
134. **Settings changes require full backend restart** — There is no hot-reload path; the engine should re-read settings on mutation via handle_command rather than needing a Render redeploy.
135. **No settings persistence layer** — User-edited settings have nowhere durable to live (SQLite table), so any UI setting would vanish on Render free-tier restarts.
136. **No settings restore on boot** — State restore covers positions but not user preferences, so alert prefs/watchlist edits would reset after every dyno recycle.
137. **No settings export to file** — Users can't download a JSON of watchlist+risk+alerts to back up before experimenting or to migrate between deployments.
138. **No settings import with schema validation** — Restoring a config backup should validate structure and value ranges rather than blindly overwriting engine state.
139. **No reset-to-defaults button** — After breaking things by tweaking, users need a one-tap revert to the shipped defaults per section, not full config.
140. **No visible display of current effective config** — The dashboard never shows what risk %, capital, or gates the engine is actually running with, so users trade blind to their own parameters.
141. **No env-vs-override provenance indicator** — When a setting can come from env, default, or user edit, the UI should show which source won, or debugging misbehavior is guesswork.
142. **Alert preferences not configurable per event type** — Users can't turn off break-even pings while keeping stop alerts, so Telegram becomes noise and real alerts get ignored.
143. **No quiet hours / do-not-disturb window** — Positional review messages fire whenever the engine wakes; users need a no-alerts window outside market hours.
144. **No per-channel alert routing setting** — No way to choose Telegram-only, dashboard-only, or both per alert class (entries vs stop hits vs daily review).
145. **No alert throttling/cooldown setting** — Repeated give-back or trail-update alerts on a choppy bar sequence need a user-set minimum gap between messages per position.
146. **No Telegram chat ID management in UI** — Re-pairing the bot after a phone change requires env edits; a settings field with a test-message button is basic.
147. **No 'send test alert' button** — Users can't verify Telegram wiring works without waiting for a live signal, a classic silent-failure mode.
148. **No timezone setting; IST assumed implicitly** — All session times (ORB window, EOD review) are presumably IST-hardcoded; a traveling user or server clock skew shows wrong timestamps with no override.
149. **No market-session time overrides** — NSE special sessions (Muhurat trading, half days) need editable open/close times or the ORB window fires at wrong times those days.
150. **No trading-holiday calendar management** — Users can't view or edit the NSE holiday list, so the engine scans (or alerts about staleness) on Holi/Diwali.
151. **No dark/light theme toggle** — A dashboard watched all day needs a theme setting persisted per user, especially for evening review sessions.
152. **Theme preference not persisted across sessions** — Even if OS-preference CSS exists, an explicit user choice should survive logout via localStorage or the profile.
153. **No currency/number formatting preference** — Indian users expect lakh/crore grouping (1,50,000); no setting means Western comma grouping everywhere, hurting scan-ability of PnL.
154. **No per-strategy enable/disable toggles** — Users can't switch off Gap-and-Go while keeping ORB, or disable RSI2 in trending regimes, without editing _STRATEGIES in code.
155. **No per-strategy parameter overrides** — ORB window minutes, Donchian lookback, EMA periods are fixed; a settings pane with validated ranges is expected for a rule engine.
156. **Strategy priority order not user-reorderable** — _STRATEGIES list order decides first-hit-wins per symbol, yet users cannot reorder priorities to reflect their conviction.
157. **No intraday vs positional master switches** — A user on vacation wants to pause intraday ideas but keep positional monitoring; needs a single toggle each.
158. **No kill switch / paused-mode setting** — A global 'stop generating new ideas but keep monitoring open positions' toggle is fundamental risk hygiene.
159. **No index-gate threshold settings** — NIFTY regime gates use hardcoded thresholds users can't loosen/tighten to match their own market read.
160. **No Chandelier trail multiplier setting** — ATR multiple for trailing is personal (2x vs 3x) and should be user-tunable with backtest-informed bounds.
161. **No give-back alert threshold setting** — The % of open profit surrendered before alerting is opinionated; hardcoding it makes the feature wrong for half of users.
162. **No breakeven-move trigger setting** — When to alert 'move stop to entry' (e.g. at 1R) should be configurable per user risk style.
163. **No default target R-multiple setting** — Users cannot change the reward:risk used to compute targets, so every idea reflects the developer's preference, not theirs.
164. **No brokerage/cost model settings** — costs.py assumptions (Groww brokerage, STT, slippage bps) aren't editable, so backtest and net-PnL figures diverge from the user's actual plan.
165. **No slippage assumption setting for backtests** — A single hidden slippage constant makes walk-forward results non-reproducible for users who trade illiquid mid-caps.
166. **No lot/qty rounding preference** — Users may want qty rounded to nearest 5/10 shares or capped by a max-notional-per-trade setting.
167. **No minimum-price / liquidity filter settings** — Users can't set 'ignore stocks under Rs 100 or below X crore turnover', a standard universe filter for NSE tools.
168. **No settings audit log** — Changing risk % mid-drawdown is exactly the moment you later want a journal entry: who changed what, when, from what to what.
169. **Settings mutations bypass command-audit path risk** — Any settings API must route through handle_command/_MUTATING so changes broadcast to the dashboard; no such command class exists yet.
170. **No confirmation step for risk-increasing changes** — Raising risk % or heat cap should require an explicit confirm dialog, since fat-fingering 10 instead of 1.0 is catastrophic.
171. **No settings change effect preview** — UI should show 'at risk 1%→0.5%, next RELIANCE idea qty drops from 40 to 20' before saving.
172. **No password change or account settings page** — The new login has no self-service password change, forcing DB surgery when credentials leak.
173. **No session timeout / remember-me preference** — Users can't choose how long login persists, awkward for a tool left open on a trading desk all day.
174. **.env.example drift from actual config keys** — With config.py recently modified, no automated check keeps .env.example in sync, so fresh deploys miss required vars and fail in degraded mode silently.
175. **No startup config sanity report** — On boot the engine should log/display which settings loaded, which fell back to defaults, and which env vars were malformed.
176. **No invalid-env-value handling with defaults** — A non-numeric RISK_PCT env value should warn and use the default, not crash or silently zero the sizing.

517. **Live distance-to-entry indicator on idea cards** — Cards don't show how far LTP is from the suggested entry, so the user can't tell if a breakout idea has already run 1.5% past its trigger.

519. **Side-by-side idea comparison view** — When three ideas fire simultaneously with capital for one, there is no comparison table ranking them by R:R, regime fit, or setup strength — the core daily decision.
520. **Idea sorting and ranking controls** — Idea cards render in emission order with no sort by risk, sector, or strategy confidence, so the best candidate isn't surfaced first.

529. **Keyboard shortcuts for buy/skip/sell** — No hotkeys (B/S/Esc) exist for the highest-frequency actions, slowing down desktop use during the 9:15–9:45 window when ORB ideas cluster.

## Platform & lifecycle (60 items)

548. **First-run setup wizard absent** — On a fresh clone there is no guided flow to enter Groww creds, Telegram token/chat ID, and capital settings — the user must hand-edit .env from a code tour, which is error-prone for a tool that sizes real trades.
549. **No .env schema validation at boot** — config.py reads os.environ without validating types/ranges, so a typo like CAPITAL=1,00,000 or a stray space in the Telegram token silently degrades or misbehaves instead of failing with a named error.
550. **No startup config summary banner** — The engine does not print an at-a-glance summary (capital, risk %, mode, which integrations are live vs degraded) at boot, so the user cannot confirm the engine is sizing trades with the intended parameters.
551. **Missing-credential errors are not actionable** — Degraded mode is silent about which specific variable is missing and where to get it (e.g. 'GROWW_API_KEY unset — create one at groww.in/api'), leaving the user guessing why no live prices arrive.
552. **No .env.example completeness check** — There is no test or CI step asserting every os.environ key used in config.py appears in .env.example, so new settings drift out of the template and fresh installs break mysteriously.
553. **No config validation CLI command** — There is no `python config.py --check` style doctor command to verify creds, token expiry, Telegram reachability, and SQLite writability before market open, when a broken config costs a trading day.
554. **SQLite schema has no migration system** — journal.py creates tables ad hoc with no schema_version table or migration runner, so any column change on an existing journal DB either crashes on boot or silently loses trade history.
555. **No journal DB backup before migration** — Nothing snapshots journal.db before schema changes or upgrades, risking irrecoverable loss of the only record of realized PnL and trade history.
556. **No app version number anywhere** — There is no __version__, git SHA, or build stamp surfaced in the API, dashboard, or Telegram /status, so the user cannot tell whether Render actually deployed the fix they pushed.
557. **Frontend/backend version mismatch undetected** — Vercel and Render deploy independently, but the dashboard never compares its expected API/schema version against the server's, so a stale frontend silently misrenders new snapshot fields.
558. **No upgrade notes or CHANGELOG** — There is no CHANGELOG.md recording behavior changes (e.g. new stop-trailing rules), so the user cannot correlate a change in live signal behavior with a specific deploy.
559. **No feature flags mechanism** — Risky new behaviors (new strategy in _STRATEGIES, new trailing logic) cannot be shipped dark or toggled off via env flag; the only rollback is a redeploy, dangerous mid-session with open positions.
